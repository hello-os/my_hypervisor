# my_hypervisor 内存管理模块设计与实现文档

## 1. 模块设计目标与在 Hypervisor 架构中的作用

内存管理模块是 `my_hypervisor` 的核心组成部分，它肩负着为 Hypervisor 自身及其未来运行的虚拟机提供可靠、安全、高效内存服务的重任。具体设计目标和作用如下：

*   **物理内存抽象与分配**：将底层裸物理内存抽象为离散的 4KB 物理页帧（Page Frame），并提供按需分配和释放的机制，避免内存碎片和管理混乱。
*   **虚拟化基石**：作为构建虚拟化环境的基础，提供 Stage 1 转换（虚拟机监视器使用的虚拟地址到物理地址的转换）功能，未来还将支持 Stage 2 转换（用于将虚拟机内部的客户物理地址转换为实际物理地址）。
*   **安全与隔离**：通过 MMU 机制，实现 Hypervisor 自身代码和数据区域的保护，并为未来不同虚拟机之间以及虚拟机与 Hypervisor 之间的内存隔离奠定基础。
*   **简化上层开发**：通过提供抽象的内存管理接口，使上层模块（如调度器、中断控制器）无需直接关注复杂的物理地址和页表操作。

## 2. 文件构成与功能职责

内存管理模块由两个文件组成：`mm.h` 和 `mm.c`，并与 `main.c` 和 `linker.ld` 紧密协作。

```text
my_hypervisor/
├── mm.h           # 1. 内存管理模块的公共定义、宏和 API 声明
├── mm.c           # 2. 内存管理模块的实际实现
├── main.c         # 3. 集成内存管理模块，负责初始化和 MMU 启用
└── linker.ld      # 4. 定义内存布局和关键符号（如 _end），供 mm.c 使用
```

### 2.1 mm.h (头文件)

*   **设计目的**：提供模块化的接口，供其他 C 文件调用，并定义内存管理相关的常量和标志。
*   **核心功能**：
    *   **常量定义**：`PAGE_SIZE`（4KB）、`PAGE_SHIFT`、`PAGE_MASK` 等，统一内存操作的粒度。
    *   **MMU 属性和标志**：定义 `MAIR_EL2` 索引 (`ATTR_DEVICE_INDEX`, `ATTR_NORMAL_INDEX`) 及其对应的属性值，以及 ARMv8 EL2 下的页表项 (PTE) 标志（如 `PTE_VALID`、`PTE_TABLE`、`PTE_PAGE`、`PTE_AF`、`PTE_SH_INNER` 等）。
    *   **高层映射标志**：提供简化的宏 `MAP_NORMAL` 和 `MAP_DEVICE`，封装常用的页表属性组合，提高代码可读性。
    *   **API 声明**：`mm_init()`、`alloc_page()`、`free_page()`、`map_page()` 和 `enable_mmu_el2()`，构成内存管理模块的公共接口。

### 2.2 mm.c (实现文件)

*   **设计目的**：实现 `mm.h` 中声明的所有内存管理功能，包括物理页分配、页表构建和 MMU 启用。
*   **核心功能**：
    *   **物理页分配器**：实现一个基于链表（Free-list）的 4KB 物理页分配器。在 `mm_init()` 中，它会根据 `_end` 符号和 `RAM_END` 定义的 RAM 范围，将所有可用的物理内存划分为 4KB 页帧，并构建一个空闲链表。`alloc_page()` 从链表中取出一个页，`free_page()` 将页归还链表。
    *   **四级页表构建**：实现了 `map_page()` 函数，该函数负责递归地分配和填充 4 级页表（L0, L1, L2, L3），将给定的虚拟地址 (`va`) 映射到物理地址 (`pa`)，并应用指定的页表标志。
    *   **MAIR_EL2 配置**：设置 `MAIR_EL2` 寄存器，将 `ATTR_DEVICE_INDEX` 映射为 Device-nGnRnE 内存类型（不可缓存），将 `ATTR_NORMAL_INDEX` 映射为 Normal Cacheable 内存类型（可缓存，Write-Back 策略）。
    *   **TCR_EL2 配置**：设置 `TCR_EL2` 寄存器，定义转换粒度（4KB）、虚拟地址和物理地址的宽度（40位），以及内存的共享域（Inner Shareable）。
    *   **TTBR0_EL2 设置**：将顶级页表（L0）的物理地址写入 `TTBR0_EL2` 寄存器。
    *   **MMU 启用**：在完成所有页表设置后，通过修改 `SCTLR_EL2` 寄存器，启用 MMU (`M` 位)、数据缓存 (`C` 位) 和指令缓存 (`I` 位)。

### 2.3 main.c (与 MM 模块的集成)

*   在 `main` 函数中，`mm_init()` 和 `enable_mmu_el2()` 被顺序调用。`mm_init()` 完成物理内存管理器的初始化和页表构建，而 `enable_mmu_el2()` 则最终激活 MMU。
*   为了调试，`main.c` 依赖 `mm.c` 中的 `mm_uart_puts` 和 `mm_print_hex` 函数进行输出，以便在 MMU 启用前后都能正常打印日志。

### 2.4 linker.ld (与 MM 模块的交互)

*   `linker.ld` 定义了 `_end` 符号，它表示 Hypervisor 代码和初始化数据段在内存中的结束位置。`mm.c` 使用这个符号来确定物理内存的起始空闲区域，从而开始分配页面。

## 3. 核心代码逻辑与目的（作为架构师的审视）

### 3.1 物理页分配器：基于空闲链表 (`alloc_page`, `free_page`)

*   **审视与目的**：这种简单链表结构非常适合裸金属环境，因为它不依赖复杂的内存管理基础设施。它能够高效地分配和回收固定大小的页。**对于 Hypervisor 初始阶段而言，这种机制是足够且可靠的。** 缺点是可能存在外部碎片，但对于只分配 4KB 页帧的场景，碎片化影响有限。未来可以考虑引入伙伴系统以更高效地管理不同大小的内存块。
*   **实现细节**：
    *   `mm_init`：遍历 `_end` 到 `RAM_END` 之间的物理内存，将每个 4KB 块作为一个 `free_page_node` 添加到 `free_page_head` 链表中。使用 `memset` 零初始化分配的页面是良好的安全实践，可防止信息泄露。
    *   `alloc_page`：从 `free_page_head` 链表中取出第一个节点并返回。如果链表为空，则表示内存耗尽。
    *   `free_page`：将释放的页重新添加到 `free_page_head` 链表头部。

### 3.2 ARMv8 EL2 4 级页表构建 (`map_page`)

*   **审视与目的**：该实现精确遵循了 ARMv8-A 架构手册中关于 Stage 1 转换的 4 级页表描述符格式。**身份映射（Identity Mapping）** 是 Hypervisor 启动阶段的常见且稳健策略，确保在虚拟内存启用后，Hypervisor 仍然能够以其原有的物理地址访问内存，避免复杂的代码重定位。对设备 MMIO（如 UART）进行正确的 `Device` 类型映射至关重要，以确保内存访问语义正确。
*   **实现细节**：
    *   `map_page`：根据虚拟地址 (`va`) 逐级解析页表索引（L0 到 L3）。如果某个级别的页表不存在，则会通过 `alloc_page()` 动态分配一个新的页表页，并将其地址写入上级页表项中，设置 `PTE_TABLE | PTE_VALID` 标志。最终，在 L3 级别，将物理地址 (`pa`) 和 `flags` 写入页表项。
    *   页表项的标志 (`PTE_VALID`, `PTE_TABLE`, `PTE_PAGE`, `PTE_ATTR_INDEX`, `PTE_AF`, `PTE_SH_INNER` 等) 均严格对应 ARMv8 架构规范。

### 3.3 MMU 相关系统寄存器配置 (`enable_mmu_el2`)

*   **审视与目的**：对 `MAIR_EL2`、`TCR_EL2`、`TTBR0_EL2` 和 `SCTLR_EL2` 的配置是开启 MMU 的关键。配置的参数（如 `T0SZ=24` 对应 40 位 VA 空间，`PS=0b010` 对应 40 位 PA 空间）与 QEMU `virt` 平台和 ARMv8 架构的典型实现相符。`dsb sy; isb` 指令对内存屏障和指令同步的运用，确保了在 MMU 真正启用前，所有页表更新都已完成，并且 CPU 指令流水线得以刷新。**这是系统从物理地址模式安全过渡到虚拟地址模式的基石。**
*   **实现细节**：
    *   `MAIR_EL2`：通过 `ATTR_DEVICE_VAL` 和 `ATTR_NORMAL_VAL` 定义，控制不同内存属性的编码。
    *   `TCR_EL2`：精细控制地址大小、页大小和缓存策略。特别是 `SH0`、`ORGN0` 和 `IRGN0` 的设置，确保了内存的一致性。
    *   `TTBR0_EL2`：指向新构建的 L0 根页表。
    *   `SCTLR_EL2`：通过设置 `M`、`C`、`I` 位，原子性地启用 MMU、数据缓存和指令缓存。

### 4.1 内存管理模块详细步骤

1.  **初始化物理页分配器** (`mm_init`)
    *   读取 `_end` 符号，计算 Hypervisor 代码结束后第一个 4KB 对齐的地址 `free_mem_start`。
    *   从 `free_mem_start` 到 `RAM_END`（`0x48000000`），遍历每一个 4KB 页面，将其作为一个 `free_page_node` 插入到 `free_page_head` 空闲链表的头部。
    *   更新 `total_free_pages` 计数。

2.  **分配 L0 根页表** (`mm_init`)
    *   调用 `alloc_page()` 分配一个 4KB 的页作为 EL2 的顶级页表（L0 Table）。
    *   如果分配失败，则打印错误信息并停止。

3.  **构建身份映射页表** (`mm_init` 调用 `map_page`)
    *   **UART MMIO 映射**：将 `0x09000000` 地址（虚拟地址和物理地址相同）映射为 `MAP_DEVICE` 类型，确保对串口的访问不被缓存，并遵循设备内存语义。
    *   **RAM 区域映射**：从 `RAM_START`（`0x40000000`）到 `RAM_END`（`0x48000000`），逐个 4KB 页面地进行身份映射，类型为 `MAP_NORMAL`。这将使整个物理内存区域在 MMU 开启后，仍可以通过相同的虚拟地址访问。

4.  **启用 MMU** (`enable_mmu_el2`)
    *   **配置 `MAIR_EL2`**：设置内存属性间接寄存器，定义 `ATTR_DEVICE_INDEX` 为 Device-nGnRnE，`ATTR_NORMAL_INDEX` 为 Normal Memory（Write-Back Read/Write Allocate）。
    *   **配置 `TCR_EL2`**：设置翻译控制寄存器，指定 4KB 页大小、40 位物理地址宽度、以及内部共享域的缓存策略。
    *   **配置 `TTBR0_EL2`**：将之前分配的 L0 根页表的物理地址加载到翻译表基址寄存器 `TTBR0_EL2` 中。
    *   **执行内存屏障**：`dsb sy; isb` 指令，确保所有页表修改都已写入内存，并且 CPU 指令流水线已清空，避免使用旧的 MMU 配置。
    *   **修改 `SCTLR_EL2`**：设置 `M`、`C`、`I` 位，正式启用 MMU、数据缓存和指令缓存。
    *   **执行指令屏障**：`isb` 指令，确保 CPU 在启用 MMU 后，能够以新的内存管理模式继续执行。

# 虚拟 CPU 调度器 (vmm_scheduler) 设计文档 - my_hypervisor

本项目针对 **ARMv8 (AArch64)** 的特权监控环境（EL2）实现了一个多任务轮转（Round-Robin）调度器。该模块对齐 **Xvisor** 的线程及 vCPU 调度模型，负责系统级虚拟 CPU、后台进程和内核高频任务之间的上下文无缝切换与执行状态流转。

---

## 1. 为什么创建该模块？（设计目的与需求）

在多核及单核虚拟化场景下，客户机操作系统（Guest OS）通常对应一个或多个虚拟 CPU (vCPU)：
1. **多客户机并发支撑**: Hypervisor 需要在单物理核心上分时复用（Time-sharing）运行多个 vCPU，从而营造独立虚拟物理机的景象。
2. **后台任务并行机制**: 监控程序在维护虚拟化状态的同时，需要运行一些异步辅助 daemon（如定时器中断管理、异步虚拟外设刷写等）。
3. **隔离性保护**: 任务切换时需要严格保存和隔离各自的寄存器、SP 堆栈状态及运行特权状态。

---

## 2. 软件模块架构与关键技术

调度器由高层就绪任务队列链表、轮转选择逻辑，以及底层的汇编级 CPU 上下文保存恢复组件协作完成。

```text
+-----------------------------------------------------------------+
|                        Ready Task Queue                         |
|  - Head -> [vCPU Task 1] -> [vCPU Task 2] -> [vCPU Task 3]      |
+-----------------------------------------------------------------+
                               |
                               | (Round-robin selection)
                               v
+-----------------------------------------------------------------+
|                     schedule() Decision Loop                    |
|  - Identifies next task from queue head                         |
|  - Puts current running task back to queue tail                 |
+-----------------------------------------------------------------+
                               |
                               | (Calls low-level switch)
                               v
+-----------------------------------------------------------------+
|                 context_switch(prev, next)                      |
|  - Saves current ARM64 GPRs & System registers to prev->regs     |
|  - Swaps SP Stack Pointer to next->regs.sp                      |
|  - Restores next task's SPSR_EL2, ELR_EL2, and GPRs             |
+-----------------------------------------------------------------+
```

### 2.1 任务描述符 (`struct vcpu_task`) 与上下文
每个可调度实体都被一个任务控制块承载：
```c
struct vcpu_task {
    int id;                      /* 唯一任务标识 */
    struct cpu_regs context;     /* CPU 寄存器上下文容器 */
    struct vcpu_task *next;      /* 单向链表指针，用于组织就绪队列 */
};
```
上下文 `cpu_regs` 镜像了 AArch64 下完整的硬件状态：
*   **通用寄存器 (x0-x30)**: 保存任务计算中间值及链接寄存器。
*   **栈指针 (sp_el2)**: 保存任务自有的独立运行栈。
*   **异常链接寄存器与程序状态 (elr_el2, spsr_el2)**: 保存任务被切走时的下一条指令执行地址和 CPU 特权字。

### 2.2 汇编级上下文切换流程 (vcpu_asm.S)
```assembly
.globl context_switch
context_switch:
    /* 1. 保存当前被切走任务的被动寄存器 */
    stp x19, x20, [x0, #144]
    stp x21, x22, [x0, #160]
    stp x23, x24, [x0, #176]
    stp x25, x26, [x0, #192]
    stp x27, x28, [x0, #208]
    stp x29, x30, [x0, #224]
    mov x19, sp
    str x19, [x0, #248] /* 保存当前物理堆栈 SP */
    
    /* 2. 直接跳转到通用恢复例程恢复新选中的任务 */
    mov x0, x1
    b vcpu_restore
```
*   **为什么这样做**：在进行 C 语言函数调用切换时，编译器遵循 ARM64 AAPCS 规范：`x0-x18` 等临时寄存器默认由调用者（Caller）负责保护。因此在 `context_switch` 函数中，我们只需要手动在堆栈和上下文中保存被调用者保护寄存器（`x19-x30`）以及系统堆栈，从而大幅缩短了任务切换的延迟。

### 2.3 任务轮转分发 (sched.c)
`schedule()` 采用 Round-Robin (RR) 时间片复用原则：
1. 提取当前队列首部的任务作为下一个要运行的 `next` 任务。
2. 将当前已经运行完毕的 `current_task` 添加回就绪队列尾部。
3. 调用底层 `context_switch` 函数执行物理寄存器态跳转。

## 3. 任务切换完整流程大纲

```text
                  +--------------------------------+
                  |    Trigger schedule() via IRQ  |
                  +--------------------------------+
                                  |
                                  v
                  +--------------------------------+
                  |  Select next = ready_list      |
                  +--------------------------------+
                                  |
                                  v
                  +--------------------------------+
                  |  Move current to queue tail    |
                  +--------------------------------+
                                  |
                                  v
                  +--------------------------------+
                  |  context_switch(&curr, &next)  |
                  +--------------------------------+
                                  |
            +---------------------+---------------------+
            |                                           |
            v (Save Old Context)                        v (Restore New Context)
  +---------------------------+               +---------------------------+
  |  Save GPRs x19-x30        |               |  Load next SPSR & ELR     |
  |  Save SP -> curr.sp       |               |  Load GPRs x0-x30         |
  +---------------------------+               |  MSR SP_EL2, next.sp      |
                                              +---------------------------+
                                                            |
                                                            v
                                              +---------------------------+
                                              |       eret (Run Task)     |
                                              +---------------------------+
```

## 4. 模块验证

由于系统初始化完成后，任务会频繁地在各个 vCPU 特权级中流转，我们通过以下两种手段交叉检验：
1. **测试进程法**: 运行两个独立循环打印不同控制台标志的任务（`Task A` 和 `Task B`），验证其输出交替出现。
2. **汇编模拟器追踪**: 使用 QEMU 进行 gdb 联调，追踪任务切换瞬间 `SP_EL2` 寄存器值确实由 `Task A` 所属的堆栈段安全偏移至 `Task B` 的段，表明栈桢切换完美闭环。
